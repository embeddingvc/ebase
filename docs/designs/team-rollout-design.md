## Problem Statement

A recruiting team does 20-50 LinkedIn candidate outreach messages per week per person. At this volume, personalization quality degrades, follow-ups slip through the cracks, and pipeline tracking falls apart. The builder created a working LinkedIn automation platform (Python + Playwright + MCP, ~4,200 lines) with 30 MCP tools and 5 Claude skills. The system works for the builder but hasn't been rolled out to the team yet. The team has a sponsor and other technical teammates who want to use it. The gap: going from "works for me" to "installable and usable by my technical teammates in Claude Code/Desktop."

## Demand Evidence

- Builder is the primary user running real recruiting outreach through the tool daily
- Team project with official sponsorship - leadership knows and backs this
- Other technical teammates have expressed interest in using it
- Medium volume (20-50 candidates/week) creates real pain: at this scale, manual personalization degrades and follow-ups slip
- The web wrapper added on May 12 signals active development toward team accessibility

## Status Quo

Before the tool: manual LinkedIn messaging at 20-50 candidates/week, tracking in spreadsheets or memory, copy-pasting connection notes that get increasingly generic, follow-ups missed because there's no pipeline state. Each recruiter is an island - no shared context on who's been contacted.

The industry: LinkedIn's own Hiring Assistant saves 4+ hours per role. SaaS tools (Dux-Soup, LinkFusion, HeyReach) charge per seat and require separate UIs. MCP-based recruiting integrations are emerging (Ashby, Crelate, Manatal all ship MCP servers). Safe daily limits: 20-25 connection requests, 40-50 DMs, 80-100 profile views.

## Target User & Narrowest Wedge

**Target user:** Technical teammates at the same company who already use Claude Code/Desktop for their work. They do recruiting outreach as part of their role (not professional recruiters). They're comfortable with CLI tools and natural language commands.

**Narrowest wedge:** A one-command install that gives teammates Claude skills for LinkedIn outreach: "connect me with linkedin.com/in/jane-doe", "follow up with everyone who accepted last week." No new UI to learn, no dashboard to check. The AI agent IS the interface.

## Constraints

- Must respect LinkedIn rate limits: 20-25 connection requests/day, 40-50 DMs/day, 80-100 profile views/day
- Requires a real Chrome browser session (CDP) - each user runs their own authenticated Chrome
- Must be installable in under 5 minutes by a technical user on macOS
- No shared state in v1 (each user operates independently)
- Account safety is non-negotiable - the tool must never put a LinkedIn account at risk

## Premises

1. The volume (20-50/week) justifies automation; teammates would adopt a working tool
2. Distribution via Claude Code/Desktop skills is the right model (not web UI/SaaS/extension)
3. Real Chrome via CDP is the right technical approach (avoids headless detection)
4. Account safety over speed (respect LinkedIn rate limits)
5. Teammates are technical and already use Claude Code/Desktop

## Cross-Model Perspective

An independent Claude subagent reviewed this session cold. Key findings:

**Steelman:** An AI-native recruiting co-pilot that turns outreach into a conversational interface. The bottleneck isn't automation alone - it's cognitive overhead. By embedding into Claude Code as MCP tools + skills, teammates get a recruiting agent that manages the full pipeline while staying under LinkedIn's radar.

**Key insight:** The decision that "the AI agent IS the interface" is the entire game. Skills can be chained, natural language captures intent better than forms, and one install command means 5 teammates adopt this week vs. 0 ever.

**Challenged premise:** Premise 5 was challenged - the subagent argued Claude Code is a developer tool and non-technical recruiters won't adopt it. The builder defended this: teammates are technical and already use Claude Code. The challenge remains valid as a future risk if the tool expands beyond the immediate team.

**Prototype suggestion:** The subagent recommended building a web UI (Next.js + SQLite) for non-technical users. This was rejected for v1 based on the team's technical profile, but remains the right move if the tool expands to non-technical recruiters.

## Approaches Considered

### Approach A: Ship What Works (CHOSEN)
Polish the install, add rate-limiting safety guardrails, ship to 2-3 teammates now. Validate team adoption before investing in more features.

- Effort: S
- Risk: Low
- Pros: Fastest path to adoption, leverages everything already built, validates the hypothesis cheaply
- Cons: No duplicate outreach prevention, no team-wide pipeline visibility, each user is an island

### Approach B: Team-Safe Multi-User
Add shared prospect index, enforced rate limits, and team pipeline visibility before launch.

- Effort: M
- Risk: Med
- Pros: Prevents duplicate outreach, enforces safety at tool level, team-wide visibility
- Cons: Shared state introduces complexity, more things to break during install, may be premature for 3-5 person team

### Approach C: Distributable Skill Pack
Package as a versioned, self-contained skill pack (like gstack) with first-run wizard and self-updating installer.

- Effort: L
- Risk: Med
- Pros: Professional distribution, built-in onboarding, sharable beyond the team
- Cons: Over-invests in packaging before validating adoption, complex to replicate gstack's model

## Recommended Approach

**Approach A: Ship What Works** because the most important unknown is whether teammates actually adopt the tool, not whether the architecture is perfect. Ship the working system with safety guardrails, measure adoption, then upgrade based on what breaks.

### Implementation Plan

1. **Polish install.sh** - Ensure one-command setup works on any Mac with Python 3.10+. Handle edge cases: Playwright browser install, CDP port conflicts. MCP registration writes to `~/.claude.json` (Claude Code) and `~/Library/Application Support/Claude/claude_desktop_config.json` (Claude Desktop), adding the `linkedin` server entry pointing to `tools/server.py`. Skills are installed globally to `~/.claude/skills/` via symlinks.

2. **Add rate limiting to MCP server** - Enforce LinkedIn safe limits at the tool level:
   - Max 25 connection requests per day per account
   - Max 50 DMs per day per account
   - Max 100 profile views per day per account
   - Configurable via environment variable or config file
   - Clear error messages when limits are hit ("Daily connection request limit reached. Resume tomorrow.")
   - State tracked in `~/.linkedin-outreach/rate_limits.json` with timestamps; day rollover uses local timezone midnight
   - State file persists across process restarts and MCP server reconnections

3. **Create `/setup-outreach` skill** - Interactive first-run wizard that:
   - Walks the user through launching Chrome with CDP
   - Guides persona.json configuration (name, title, company, tone)
   - Tests the MCP connection with a simple profile scrape
   - Confirms everything works with a "You're ready!" message

4. **Harden existing skills for multi-user safety** - Review each skill for assumptions that break with multiple users:
   - Each user gets isolated state at `~/.linkedin-outreach/{username}/` for logs, prospects, conversations, and config
   - Chrome CDP URL configurable per user: env var `CDP_URL` takes precedence over `~/.linkedin-outreach/config.json` `{"cdp_url": "http://localhost:9222"}`, defaults to `http://localhost:9222`
   - Persona config must be per-user, not shared
   - Acceptance test: install on 2 different Macs (or user accounts) and verify both can operate independently without file or state conflicts

5. **Write team onboarding README** - 1-page doc with:
   - Prerequisites (Python 3.10+, Chrome, Claude Code)
   - Install command (`./install.sh` or `curl | bash`)
   - "Try it now" example: "Run `/send-connection-request linkedin.com/in/john-doe` in Claude Code to send your first AI-personalized connection request"
   - Common troubleshooting (CDP connection refused, MCP server not registered, Playwright browser missing)

6. **Ship to 2-3 teammates** - Hand the README to two teammates. Watch them install and use it. Note every friction point. Fix the top 3 before wider rollout.

## Error Handling & Recovery

- **Chrome crash / CDP disconnect:** MCP tools should catch connection errors and return a clear message: "Chrome connection lost. Restart Chrome with `make browser` and retry." No silent failures.
- **LinkedIn UI change:** Playwright selectors may break. The tool should fail loudly with the selector that failed, not silently skip actions. Mock mode (`tools/mock.py`) provides a fallback for testing when LinkedIn UI changes.
- **Network failure mid-operation:** Retry once after 5 seconds. If still failing, return error to the user. Never leave a prospect in an inconsistent state (e.g., connection sent but not logged).
- **Session expiration:** Before each LinkedIn action, check if the session is valid (e.g., verify a basic page load). If expired, return: "LinkedIn session expired. Log in again in Chrome and retry."
- **Rollback plan:** If critical issues emerge during teammate pilot, pause rollout and document blockers in a `KNOWN_ISSUES.md` before proceeding.

## Open Questions

- Should there be a shared "do not contact" list to prevent duplicate outreach even without shared state? (Deferred to Approach B, but worth discussing with the team before launch.)
- What metrics should be tracked to measure team adoption? (Daily active users, messages sent per person, skill invocation frequency)

## Resolved Questions

- **CDP ports:** Each user on the same machine must use a unique CDP port (9222, 9223, etc). Different machines can reuse the same port. Configurable via `CDP_URL` env var.
- **Session expiration:** Handled by pre-action session health check (see Error Handling above).

## Success Criteria

- 2+ teammates besides the builder actively using the tool for real outreach within 2 weeks of rollout
- Install time under 5 minutes for a fresh macOS setup
- Zero LinkedIn account restrictions caused by the tool
- At least 20 outreach messages per week per user through the tool (matching or exceeding manual volume)
- Positive qualitative feedback: "this is faster than doing it manually"

## Distribution Plan

- One-command install via `install.sh` (or `curl | bash` from a private repo)
- Registers MCP server with Claude Code/Desktop
- Installs skills to `~/.claude/skills/` (or project-local `.claude/skills/`)
- Updates: git pull + re-run install.sh (manual for v1; automated in future Approach C)

## Dependencies

- Teammates need macOS with Python 3.10+ and Claude Code installed
- Each user needs their own LinkedIn account and Chrome browser
- Team needs to agree on outreach norms (daily limits, messaging tone, who contacts whom)

## The Assignment

Don't help unless they're stuck for more than 2 minutes. Write down every moment they hesitate, every error message they hit, every question they ask. That list IS your v1.1 roadmap.
